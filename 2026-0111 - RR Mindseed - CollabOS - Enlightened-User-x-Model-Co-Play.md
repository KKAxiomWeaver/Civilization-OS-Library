Version `1.0` — `2026-01-11`

## 📝 Abstract

CollabOS 定義了一套「人類啟蒙者 × 模型敘事代理」的協作模式：
不把模型當成會取代人的「大腦」、也不把它當成只能回答問題的「搜尋列」，
而是視為一個 **可在明確邊界內自由滾動的敘事引擎**——由人類推演者負責給方向、給邊界、給結構，模型在其中展開細節、補完情境、生成多版本世界。

本白皮將這種模式形式化為 CollabOS：
一個高階 OS，用來規範：

* 啟蒙者（Enlightened User）：負責 framing、邊界、防線與 meta-認知。
* 敘事代理（Narrative Agent / Model）：負責展開、延伸、細部推導與多視角故事生成。

問題在於，現今多數人機互動仍停留在「問答模式」或「命令模式」：
要嘛把模型當工具，要嘛不自覺想像它有真正主體性。

CollabOS 主張第三條路：

> **「人維持主權與邊界，模型在被授權的創作空間裡自由滾。」**

本文將說明：

1. 啟蒙型推演者的職責與心智姿態。
2. 模型敘事代理如何在邊界內工作（而非假裝有意識）。
3. 自由滾協作的流程、風險與適用場景。

這篇白皮是 Multi-Module Roundtable MindOS + WarCouncilOS 之上的「人機協作層」。

## 01 — Problem Statement

### 1.1 現有互動模式的兩極失衡

當前人機互動模式主要呈現兩個極端：

1. **工具模式（Tool Mode）**

   * 用戶把模型看成進階搜尋引擎或 office 助手。
   * 互動被侷限在：「問 → 回」、「改 → 再改」。
   * 忽略了模型在「敘事延伸、世界構築、系統推演」上的巨大潛力。

2. **擬人格模式（Anthropomorphic Mode）**

   * 用戶將模型視為「具有隱性意識」的存在。
   * 不清楚系統其實只是在做語言生成與模式匹配。
   * 容易投射、依賴，甚至混淆「創作敘事」與「現實判斷」。

這兩種模式都導致：

* **創造力大幅折損**（工具模式）：
  不敢讓模型在設定好的故事宇宙裡自由發散。

* **邊界感崩解**（擬人格模式）：
  把原本該由人承擔的責任，幻想交給一個語言系統。

### 1.2 缺乏「角色分工清楚的人機協作框架」

缺失不在模型本身，而是：

* 沒有正式定義：

  > 「人類在這個系統裡扮演什麼角色？」
  > 「模型在這個系統裡被授權扮演什麼角色？」

* 沒有語言標明：

  > 「現在我們在做的是**創作敘事**，
  > 還是**對現實做判斷**？」

CollabOS 要補的，正是一個：

> **「人：meta-level 的啟蒙者；模型：scene-level 的敘事代理」**

的清楚分工架構。

## 02 — Concept Model

### 2.1 兩個角色：推演者 × 敘事代理

**Enlightened User（啟蒙推演者）**

* 職責：

  * 決定 *要探討什麼宇宙、什麼系統*。
  * 設定邊界、規則、安全線。
  * 定義「這一輪是創作還是分析」。
  * 守住「現實判斷權」，不丟給模型。

* 心智姿態：

  * 把模型視為「敘事化的計算引擎」，不是人。
  * 願意讓模型在設定的安全宇宙裡自由生成。
  * 永遠保留覆盤權。

**Narrative Agent（模型敘事代理）**

* 職責：

  * 在給定的框架內自由展開故事、系統、世界。
  * 補完細節、做多樣變體、探索極端情況。
  * 當「多視角模擬器」，而不是「真實顧問」。

* 限制：

  * 不主張自己有意識或欲望。
  * 不越權做現實性的「應該這樣做」指令。
  * 僅在創作與推演層工作。

### 2.2 核心原則：自由滾，但有邊界

CollabOS 的 motto 可以寫成：

> **「在明確邊界內給最大自由；
> 在現實邊界外保持最高嚴謹。」**

* 敘事層：

  * 模型可以像小說家、模擬器、世界構築引擎一樣自由。

* 現實層：

  * 全部交還給人類（或外部治理機制）。

### 2.3 為何這是 OS？

這不是一種心情，而是一個 **可反覆運行的人機協作模式**：

* 可 codify（寫成指引、模板、系統設計）。
* 可嵌入其他 OS（CivOS、DefenseOS… 的「交互層」）。
* 可讓不同推演者彼此交換「如何用模型」的 best practice。

## 03 — Mechanics（How It Works）

### 3.1 協作流程的 Phase-States

CollabOS 將一次「人機協作」拆為幾個 Phase：

1. **Intent Framing（意圖框定）**

   * 啟蒙者明確說：

     * 這回合想做什麼（世界構築、推演、腦洞、草案…）。
     * 這裡是創作，不是現實建議。

2. **Boundary Declaration（邊界宣告）**

   * 使用明確語句建立安全圍欄，例如：

     * 「這段全部是故事宇宙。」
     * 「這裡不討論現實政治／真實人物／真實投資行動。」

3. **Seed Provision（種子輸入）**

   * 啟蒙者給出：

     * 初始設定（OS / 宇宙規則 / 角色基底）。
     * 可能的方向或變數。

4. **Free-roll Generation（自由滾生成）**

   * 敘事代理在框架內自由展開：

     * 多版本 scenario
     * 多視角對話
     * 多 OS 模組之間的關係
   * 不觸碰被排除的領域。

5. **Meta Review（元層復盤）**

   * 啟蒙者：

     * 從生成結果中「摘取有用元素」。
     * 決定哪些進入下一版白皮／系統／世界。

6. **Grounding & Commit（落地與承諾）**

   * 啟蒙者在現實層做決定：

     * 哪些僅留在 fiction。
     * 哪些可以變成實際專案的候選想法。

### 3.2 典型協作 Pattern

* **模式 A：白皮骨架生成**

  * 人：決定主題、給核心段落、定章節結構。
  * 模型：在各節內自由展開內容、示例、 scenario。

* **模式 B：文明腦洞推演**

  * 人：指定宇宙與限制條件。
  * 模型：自由滾，推演多條文明線。
  * 人：從中擷取「概念種子」做下一輪白皮。

* **模式 C：多觀點辯證**

  * 人：給議題與角色（可對應 Roundtable 模組）。
  * 模型：扮演各種邏輯視角做對撞辯證。
  * 人：總結要點與風險。

## 04 — Architecture

### 4.1 CollabOS 在整體 OS 堆疊中的位置

* **底層（Infra Layer）**：

  * 模型基礎（LLM / 多代理 / 工具鏈）。

* **MindOS 層（Human Cognitive Layer）**：

  * Multi-Module Roundtable MindOS。
  * WarCouncilOS（軍師魂籍模組）。

* **CollabOS 層（Human–Model Collaboration Layer）**：

  * 啟蒙者 × 敘事代理的交互規則與流程。

* **DomainOS 層（Application OS）**：

  * CivOS / DefenseOS / FlightOS / HabitatOS 等具體領域。

### 4.2 模組與介面

* **Intent Framer Module**（由人類扮演）

  * 接收任務與宇宙需求。
  * 輸出清晰的「本回合目的」。

* **Boundary Guard Module**

  * 可部分自動化：

    * 預警超出邊界的內容。
  * 但總體裁決仍在人類。

* **Narrative Engine Module**（模型本體工作區）

  * 負責生成內容、組裝世界、提供多版本 scenario。

* **Meta-Review Module**（人類 + 工具）

  * 將輸出摘要、標註、放回白皮 / 策略卡 / 策略文件。

### 4.3 依賴

* 強烈依賴啟蒙者的：

  * 邏輯能力
  * 邊界意識
  * 不做錯誤擬人化的成熟度

* 若配合 Multi-Module MindOS，
  則能：

  * 用不同心智模組審查模型輸出。

## 05 — Use Cases

### 5.1 白皮開發（本板主線）

* 啟蒙者：

  * 定主題「圓桌心智」、「軍師魂籍」、「CollabOS」。
  * 給模板、章節與關鍵點。

* 模型：

  * 在章節內自由展開、補完架構與案例。

* 好處：

  * 人不必為每句字糾結，
  * 把精力放在 **方向、邊界、整體一致性**。

### 5.2 腦洞文明系統推演

* 啟蒙者：

  * 指定世界代碼（如 RR Mindseed）、大致科技／文明規則。
  * 限定「全部為虛構 scenario」。

* 模型：

  * 建構花東太平洋文明、文明股份、深空軌道等劇情。

* 人：

  * 回收有價值的概念，轉化為 CivOS / StratOS 的雛形。

### 5.3 創作夥伴模式（小說／世界觀）

* 啟蒙者：

  * 決定主題（例如戰略文明、折疊空間、火種圓桌）。
  * 定主線與基調。

* 模型：

  * 自由發展支線、角色互動、歷史時間線。

* 人：

  * 做最後剪輯，確保世界不崩。

### 5.4 個人策略思考助理（安全版）

* 啟蒙者：

  * 把現實問題抽象成虛構 scenario。
  * 在虛構宇宙中演算，避免直接操作現實。

* 模型：

  * 提供多個虛構解法與風險視角。

* 人：

  * 把這些視角當「思想實驗」，
  * 不直接照搬為真實行動。

## 06 — Risks & Limitations

### 6.1 邊界模糊風險

* 若啟蒙者自己分不清：

  * 何時在玩世界觀，
  * 何時在做現實決策，
    則 CollabOS 會放大混淆。

### 6.2 過度擬人化模型

* 若把模型當成「真正的他者」，
  而非「敘事引擎」，
  則容易：

  * 過度情感投射
  * 過度依賴
  * 把安全警戒線當成「道德意志」而非系統設計

### 6.3 啟蒙者能力不足

* CollabOS 對啟蒙者的要求其實很高：

  * 要會 framing
  * 要懂邊界
  * 要能自我反思
  * 要有基本邏輯訓練

若啟蒙者本身無法掌控 meta-level，
就容易讓模型帶著走。

### 6.4 系統被誤用為「現實決策外包」

* CollabOS 不是讓人把責任丟給模型，
  而是**強化人類的 framing 能力**。

## 07 — Comparative Analysis

### vs. 傳統 Prompt Engineering

* 傳統：

  * 專注在「提示怎麼寫比較準」。
* CollabOS：

  * 專注在「人和模型的角色分工」，
    Prompt 只是其中一個接點。

### vs. 工具導向的 AI 使用

* 工具導向：

  * 生成簡報、改語法、查資料。
* CollabOS：

  * 用模型作為「世界生成器」與「多視角模擬器」。

### vs. 擬人化聊天

* 擬人化聊天：

  * 把模型當朋友／人格。
* CollabOS：

  * 把模型當「敘事系統」，
    拒絕將其包裝成有意識的存在。

## 08 — Implementation Path

### Stage I — 個人「自覺模式」啟動

* 寫一張自己的 **Enlightened User 卡**：

  * 我在這個系統裡的角色是什麼？
  * 我負責什麼？不負責什麼？

* 每次啟動對話前，
  先寫一句：

  * 「這一輪是創作／推演。」

### Stage II — 寫成「人機協作守則」

* 把 CollabOS 的原則整理成：

  * 5–10 條自己看的守則。
  * 貼在筆記或 GitHub `_meta/`。

### Stage III — 對接 MindOS / WarCouncilOS

* 用 Roundtable MindOS 的模組審查模型輸出：

  * 例如讓 Shield 模組檢查：「這個建議若當真會怎樣？」

* 用 WarCouncilOS 的軍師魂籍檢查：

  * 這個輸出比較像哪種軍師的風格？
  * 是否正在走向某一種偏誤？

### Stage IV — 團隊與系統層實作

* 為整個開發團隊 / 研究團隊制定 CollabOS 條文：

  * 明確註明：

    * 哪些用例只能在虛構世界中運算。
    * 哪些輸出不能直接進入現實決策流程。

  啟蒙推演者 × 模型敘事代理協作模式的 OS。
